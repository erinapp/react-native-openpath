OpenpathMobile

The SDK that provides access to Openpath functionality for iOS-based mobile devices.

Important

If viewing this documentation in XCode, be sure to choose “Swift” as the language, both in the sidebar and above.

Overview

Adding the SDK to your project

The SDK is provided as a binary framework in a CocoaPods package, delivered in a .zip file. The package also includes binary frameworks for all closed-source dependencies of the SDK.

To integrate the SDK:

  • Set up CocoaPods for your app, if it is not already. Refer to CocoaPods documentation for details.
  • Enable dynamic frameworks in your Podfile: ruby use_frameworks!
  • Unzip the SDK archive in a known location.
  • Add the following to your Podfile in the section for your app target, filling in the correct SDK directory: ruby pod 'OpenpathMobile', path: '/sdk/directory'
  • Add a post_install hook to your Podfile as needed, as shown below and in the Example app.
  • Ensure your app is using Swift 5.x
  • Enable the “Build Libraries for Distribution” setting as needed (see below)
  • From the terminal, run pod install
  • Build and run your app within Xcode

The “Build Libraries for Distribution” setting.

Since the SDK is distributed in binary form, the SDK’s dependencies must be built with the “Build Libraries for Distribution” build flag set to Yes. This tells Xcode to build those dependencies in such a way that the SDK can call into them without needing to be recompiled.

For a simple app, this can be accomplished with the following steps:

  • In Xcode, select the build settings for your app target, and chose Yes for “Build Libraries for Distribution”
  • From the terminal run (or re-run) pod install
  • Cocoapods will apply that build setting to all Pod projects, including OpenpathMobile and all its dependencies.

In a more complex app, you may not want to apply that setting to all Pod projects. For example, this can cause problems with some React Native libraries. In that case, here’s some code you can add to your Podfile that will apply the setting only to OpenpathMobile’s dependencies:

  • Add this somewhere near the top of the Podfile:
def get_dependencies_recursively(installer, root)
  top_dependencies = installer.pod_targets.find { |pt| pt.name == root }.dependencies
  dependencies = top_dependencies.to_set
  top_dependencies.each do |dep|
    dependencies.merge(get_dependencies_recursively(installer, dep)) if dep != root
  end
  dependencies.to_a
end

def openpath_postinstall(installer)
  openpath_dependencies = get_dependencies_recursively(installer, 'OpenpathMobile')
  installer.generated_projects.each do |project|
    project.targets.each do |target|
      next unless openpath_dependencies.include?(target.name)

      puts "Target #{target} is an Openpath dependency, so setting BUILD_LIBRARY_FOR_DISTRIBUTION"
      target.build_configurations.each do |config|
        config.build_settings['BUILD_LIBRARY_FOR_DISTRIBUTION'] = 'YES'
        # Some depended-on Pods set this to an obsolete iOS version
        config.build_settings['IPHONEOS_DEPLOYMENT_TARGET'] = '12.0'
        # None of the Openpath dependencies need to be signed
        config.build_settings['CODE_SIGNING_ALLOWED'] = 'NO'
      end
    end
  end
end
  • Add a post_install block to the Podfile, or add to your existing one:
  post_install do |installer|
    openpath_postinstall installer

    # Any other post_install scripts go here
  end

  • Re-run pod install

  • Reload the app in Xcode and rebuild

Enabling background modes and privacy keys

The Openpath SDK requires the ability to access Bluetooth and location while the app is in the background. (The user will be prompted to approve these permissions interactively on first use.)

Ensure your app’s Info.plist file contains the following background modes:

<key>UIBackgroundModes</key>
<array>
    <string>bluetooth-central</string>
    <string>location</string>
</array>

Openpath requires that the user has the following settings on their phone:

  1. (Global) Bluetooth -> ON
  2. (Global) Location Services -> ON
  3. (optional) (Global) Fitness Tracking -> ON
  4. Openpath Bluetooth Permission -> Enabled
  5. Openpath Location Permission
  6. ALWAYS - for Wave/Touch to unlock reliably
  7. WHEN IN USE - for unlocks only when the app is opened
  8. (optional) Openpath Motion & Fitness Permission -> Enabled

Bluetooth and Location Services are used to communicate with Openpath readers, to ensure that the app properly wakes up when near readers, and to allow the Wave feature to work reliably. Openpath never tracks or stores a user’s location information for any other purpose. Motion & Fitness is used to help save battery and reduce data usage, as Openpath will turn off unnecessary activity if the phone is stationary.

Include the following privacy keys in your Info.plist so that when a user is prompted for permission, the given text is displayed. The strings below are examples, and you may re-word them as you wish.

<key>NSBluetoothAlwaysUsageDescription</key>
<string>We need access to Bluetooth to discover and unlock nearby entries.</string>
<key>NSBluetoothPeripheralUsageDescription</key>
<string>We need access to Bluetooth to discover and unlock nearby entries.</string>
<key>NSLocationAlwaysAndWhenInUseUsageDescription</key>
<string>
  We need "Always" location permission in order to discover nearby entries when the app is
  closed. This allows you to unlock via "Touch" without opening your phone. We do not store,
  log, or share your location data.
</string>
<key>NSLocationAlwaysUsageDescription</key>
<string>
  We need "Always" location permission in order to discover nearby entries when the app is
  closed. This allows you to unlock via "Touch" without opening your phone. We do not store,
  log, or share your location data.
</string>
<key>NSLocationWhenInUseUsageDescription</key>
<string>
  We need "While Using App" location permission in order to discover nearby entries. We do not
  store, log, or share your location data.
</string>
<key>NSMotionUsageDescription</key>
<string>
  We use motion to turn off services when your phone is stationary in order to save battery
  and reduce data usage.
</string>

For iOS14, Apple added the ability for location accuracy to be reduced or approximated. In order for unlocks to be verified as “on-site” and near a reader according to geofence rules, the SDK needs full location accuracy, so this setting must be off. The following can be added to your Info.plist file, though the default value is false if this key is not included.

<key>NSLocationDefaultAccuracyReduced</key>
<false/>

Beacon Region Monitoring

The Openpath SDK uses beacon region monitoring so the SDK can wake up and start services when the device comes within range of readers.

Apple allows for a maximum of 20 beacons regions to be monitored per app. Openpath monitors 11 regions, so there are 9 remaining regions available for your use.

In order to help prevent region identifier collision, Openpath regions are prefixed with op-beacon.

Using the OpenpathMobileAccessCore class

Once you’ve added the framework to your project, updated Info.plist, and installed the CocoaPods libraries, you are ready to write some code.

The OpenpathMobileAccessCore class is a singleton, available through the shared static variable.

Set your class as the SDK’s delegate:

class MyClass: OpenpathMobileAccessCoreDelegate {
  init() {
    OpenpathMobileAccessCore.shared.delegate = self
  }
}

IMPORTANT: Always access OpenpathMobileAccessCore.shared on the main thread.

iOS Example App

The iOS example app uses the Openpath Mobile Access Core SDK to:

  • Provision one or more user mobile credentials onto the app
  • Switch between users on the app
  • View the list of entries that the user has access to
  • Unlock an entry
  • Unprovision all the users or select one to unprovision from the list of inactive users

The Example App targets iOS 15+ and uses advanced features that may not be available in previous iOS versions.

For the purposes of this demo, we will show provisioning through two different ways.

  1. The app can be provisioned via email and password by allowing a user to log in and generate their own mobile setup token to use for provisioning. Usually, only a user with admin rights to the Openpath Control Center (admin portal) has a password.

  2. The app can be provisioned via mobile setup token. In practice, most users won’t have an Openpath password. We recommend that you create a service user (super admin) in the Openpath Control Center, and use that service user in your cloud to make Openpath API calls to generate a temporary mobile setup token for a particular user’s mobile credential. Then this token should be passed from your cloud server to the mobile app and into the “provision” SDK function.

For more information on the provisioning workflow, please refer to the Openpath Mobile Access SDK README section entitled “Provisioning Workflow”.
