Release v0.3.0

Release Notes

SDK

  • Improvement - app wakeup improved, especially for first unlock of the day.
  • Update - Customer can only use nine beacons (instead of ten) (we are now using one more for improved app wakeup).
  • API Update - replaced completion handler in four functions, provision, switchUser, syncUser, and unprovision, by four new delegates.
  • API Update - added four delegates onProvisionResponse, onSwitchUserResponse, onSyncUserResponse, and onUnprovisionResponse.

Sample App

  • Feature - added setup mobile token provisioning

Migration Guide

  • As part of improving app response times, Openpath now monitors 11 instead of 10 regions. So now there are 9 (instead of 10) remaining regions available for your use.
  • JWTDecode version upgraded to 2.6

switchUser function & new onSwitchUserResponse delegate

The switchUser function’s completion callback was removed so the function now looks like this:

func switchUser(userOpal:String)

The switchUser response is now handled via a delegate function:

func openpathMobileAccessCore(_ openpathMobileAccessCore: OpenpathMobileAccessCore, onSwitchUserResponse message: [String: Any])

The message payload returned in the delegate function remains the same and did not change from the message that was in the completion callback.

The purpose of this change is to standardize asynchronous requests so that its responses are handled via a delegate under the template name on + name + `response.

syncUser function & new onSyncUserResponse delegate

The syncUser function’s completion callback was removed so the function now looks like this:

func syncUser()

The syncUser response is now handled via a new delegate function:

func openpathMobileAccessCore(_ openpathMobileAccessCore: OpenpathMobileAccessCore, onSincUserResponse message: [String: Any])

The message payload returned in the delegate function remains the same and did not change from the message that was in the completion callback.

The purpose of this change is to standardize asynchronous requests so that its responses are handled via a delegate under the template name on + name + `response.

unprovision function & new onUnprovisionResponse delegate

The unprovision function’s completion callback was removed so the function now looks like this:

func unprovision(userOpal:String?)

The unprovision response is now handled via a new delegate function:

func openpathMobileAccessCore(_ openpathMobileAccessCore: OpenpathMobileAccessCore, onUnprovisionResponse message: [String: Any])

The message payload returned in the delegate function is not the same, the new message payload returned contains three objects, in the previous version, the response was an empty dictionary. The three parameters are the userOpal of the user unprovisioned, see #1, and two more parameters, reasonCode and reasonDescription.

The purpose of this change is to standardize asynchronous requests so that its responses are handled via a delegate under the template name on + name + response.

The second purpose is to open a channel between the SDK client and the SDK in cases where the unprovision is triggered by the SDK. One such case is when the user’s mobile credential that was provisioned on this device is provisioned on another device (mobile credentials may only be used on one device only). In this scenario, the onUnprovisionResponse delegate will be called to indicate that the user was unprovisioned from this device.

Note: In cases where the SDK/client SDK unprovisioned all the users, the userOpal object will be null
