Release v0.4.0

Summary Release Notes

  • ACU configurations and user permissions changes no longer pushed to app (changes to your app may be required)
  • User settings changes no longer pushed to the app (changes to your app may be required)
  • State-related item data has moved (changes to your app may be required)
  • Allegion wireless locks
  • Internationalization
  • EU instance
  • New example app
  • New React Native wrapper and example app

Release Notes

ACU configurations and user permissions changes no longer pushed to app

ACU (Access Control Unit or Controller) configurations (such as entry settings, schedules, reader settings, etc.) and user permissions changes are no longer automatically pushed to the SDK when admins make these changes in the web portal (Control Center). Any updates in the portal used to be delivered via the openpathMobileAccessCore(_:onItemsSet:) and openpathMobileAccessCore(_:onItemsUpdated:) delegate callbacks automatically, but you will no longer receive these right after changes.

In order to get updates for ACU configurations and user permissions changes, we have added a new function softRefresh() to get new refreshed data from the cloud. Updates will then be delivered via the onItemsSet and onItemsUpdated delegate callbacks.

Our suggestion is to use this function when users would like to refresh their data, for example, via a “pull-down-to-refresh” feature, or a refresh button in the app. In our new example app, we have included calling softRefresh when user pulls down to refresh on the list of items screen. Note that calling softRefresh on its own is not needed on app start because it is automatically handled once all the services are initialized.

Please refer to the documentation for those functions and callbacks for more details.

Background: This change was made in order to scale the system and optimize, especially for larger organizations where configuration changes are constantly being made. This change reduces the data that is sent to the app, therefore increasing the performance and speed of the app itself. Most of the time, a manual call to refresh data is not needed because the app will naturally shut down after awhile and re-open in the background around readers, thus refreshing its data when the app restarts.

User settings changes no longer pushed to the app

In order to get updates for user settings (user cloud keys and whether user has remote or override unlock capability), we have added a new function refreshUserSettings() to get new refreshed data from the cloud. Updates will then be delivered via the openpathMobileAccessCore(_:onUserSettingsSet:) delegate callback.

Please refer to the documentation for that function and callback for more details.

State-related item data has moved

We have moved some of the state-related item data delivered from these delegate callbacks: \ - openpathMobileAccessCore(_:onItemsSet:) \ - openpathMobileAccessCore(_:onItemsUpdated:)

into the new callback: \ - openpathMobileAccessCore(_:onItemStatesUpdated:)

To trigger this new delegate callback, call the new function for a specific item: \ - refreshItemState(itemType:itemId:)

These are the fields that have moved: \ isScheduledTouchAllowed \ isScheduledUnlocked \ isScheduledNoAccess \ isScheduledRemoteUnlockAllowed \ isScheduledOverrideAllowed \ isScheduledAvailableRevert

This state data can only be queried for a single item at a time (using the itemType and itemId parameters), so the data would be best used on an drilled-down item detail screen. This new function and delegate callback are included in our new example app and state data is displayed on the item detail screen.

Please refer to the documentation for those functions and callbacks for more details.

Allegion wireless locks

This version of the SDK supports unlocking Allegion wireless locks. No additional app code is needed. If the provisioned user has access to Allegion wireless locks, these locks will be available in the list of items provided in onItemsSet. Allegion wireless locks can be unlocked by calling the unlock function, similar to the way other entries are unlocked.

Internationalization

The SDK now supports English, French, German, Italian, and Spanish.

No additional app code is needed. The SDK looks at the phone’s locale settings and returns a corresponding language string prefixed by localized for relevant fields that contain English strings.

For example, the onUnlockResponse currently includes the following fields: \ tip \ description \ result

and now includes 3 additional fields: \ localizedTip \ localizedDescription \ localizedResult

EU instance

Since the last SDK was released, we have added a European instance of the Control Center (https://control.eu.openpath.com) for orgs that are in Europe. This SDK supports provisioning users that are on the EU instance.

No additional app code is needed. As long as you are using our EU instance API (https://api.eu.openpath.com), the setupMobileToken that you generate and send to the provision(setupMobileToken:callback:) function will contain the relevant information needed for the SDK to provision the user properly.

New example app

The previous iosSampleApp has been replaced with a redesigned ExampleApp, that follows current SwiftUI best practices.

New React Native wrapper and example app

We have introduced a new React Native wrapper v1.0.0 and a React Native example app. It is available to download in a separate zip file (v1.0.0-react-native.zip).

Migration Guide

This version of the SDK relies on CocoaPods to manage dependencies. So if you haven’t already, install CocoaPods and integrate it with your project.

If you have an old version of the SDK installed, delete the old framework files and remove the references to them from your project in XCode.

Note that the exported framework has been renamed from OpenpathMobileAccessCore to OpenpathMobile so you will need to update your import statements.

Then follow the steps outlined in the [Adding the SDK to your project] section of this SDK documentation.
